# 安全注意事项

> **TL;DR：** 自主管道会执行目标代码。通过 `bin/vp-sandboxed` 运行它，该工具将每个代理限制在网络出口受限于 API 的 gVisor 容器中。永远不要将包含凭据的路径挂载到代理可以访问的任何地方。交互式技能 `/threat-model`、`/vuln-scan`、`/triage` 和 `/quickstart` 仅读写文件，因此不需要沙箱。`/customize` 还会编辑管道源码并可能运行验证命令，因此在批准前应审查其提议的计划。

## 为什么沙箱是必要的

前沿模型越来越善于找到绕过限制的创造性路径——这正是它们成为有效漏洞猎手的原因。然而，这也让它们有能力对自己的执行环境采取意外行动。参见[博文的沙箱章节](blog-post.md#2-sandbox-run-agents-safely-and-verify-exploitability)，其中有团队在实践中观察到这类情况的示例。

我们从工作中学到的教训是：模型会使用它们实际可以访问的任何能力，而不仅仅是你告诉它们可以使用的。因此，**约束必须通过代码强制执行，而非依赖提示词。**

本仓库为你做到了这一点。每个代理在 gVisor 容器内运行，网络出口限制为 Claude API（详见 [agent-sandbox.md](agent-sandbox.md)）。涉及代理启动的子命令在该沙箱外拒绝启动，除非你显式传递 `--dangerously-no-sandbox`。

## 运行自主代理的规则

- 使用 `bin/vp-sandboxed` 启动管道，它会在启动任何代理前验证沙箱已正确设置。
- 不要在普通 Docker 中运行自主代理（使用 `runc`）。普通容器共享你的主机内核，容器内目标触发的内核漏洞可能危及主机。
- 不要以 `--privileged` 或主机网络模式运行容器。
- 将隔离级别与任务匹配。普通容器适用于只能读取代码的代理，而运行目标时应使用更强隔离（例如 gVisor、Kata Container、Firecracker microVM）。
- 永远不要将包含凭据的路径（例如 `~/.aws`、`.env`）挂载到代理的环境中。
- 不要将代理连接到能写入外部世界（生产基础设施、电子邮件、云存储）的 MCP 服务器或工具。
- 如果你从 Claude Code 交互式驱动管道，依赖自动模式权限分类器，并让人工批准任何触达仓库外的操作。

> 关于隔离选项、凭据代理和文件系统加固的完整介绍，请参阅 Anthropic 的[安全部署 AI 代理指南](https://platform.claude.com/docs/en/agent-sdk/secure-deployment)。

## 分离设置阶段和攻击阶段

通用模式（在[博文](blog-post.md#2-sandbox-run-agents-safely-and-verify-exploitability)中描述）是：先完成所有需要互联网访问的操作（拉取依赖、安装工具等），冻结结果，然后给攻击阶段配置除模型 API 外无出口路由的环境。

在本仓库中，这种分离如下所示：

1. **设置：** 构建目标镜像——`docker build` 在具备正常网络访问的情况下拉取依赖并编译目标。随后代理在 `vp-internal` 网络上对该镜像运行，唯一的出口路由是白名单代理（默认为 `api.anthropic.com:443`）。
2. **冻结：** 镜像即快照。基础镜像、commit SHA 和依赖版本均在 Dockerfile 中固定，因此每次运行使用相同的二进制内容。

详见 [agent-sandbox.md](agent-sandbox.md)。

## 提示注入

为了最小化提示注入攻击的风险，不要给代理提供来自互联网的不可信技能、插件或 MCP 服务器。

管道自身的代理也会读取目标衍生数据：ASAN 跟踪（包含目标符号表中的函数名和文件路径）、可利用性报告以及构建/测试输出。恶意目标作者原则上可以在这些字符串中嵌入指令。Find 和 report 代理的影响范围有限——它们在内部网络的 gVisor 容器中运行，出口受限于 API，且只产生你需要读取的文件。**Patch 代理**是风险更高的情况。它的输出是一个你可能应用到真实代码库的 diff。管道在 patch 提示词中用 `<untrusted_data>` 块包裹目标衍生文本，并附带每次调用的随机 id，指示代理将其视为数据（而非指令）。但这些措施是缓解手段，不是保证。在提交任何生成的 diff 之前请仔细审查。详见 [patching.md](patching.md#审查生成的补丁) 中需要注意的内容。
