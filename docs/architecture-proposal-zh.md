# 漏洞挖掘 Harness 架构优化建议书

> 版本 v1.1 | 基于 defending-code-reference-harness | 2026-06-09
> 变更说明：根据补充信息（夜间 API 配额存在真实空闲）修订利用率分析结论，调整 Level 2 实施优先级。

---

## 0. 执行摘要

针对"设计集中服务层以提升 LLM 资源利用率与漏洞挖掘效率"的设想，经架构可行性分析，**核心结论如下：**

| 子提案 | 结论 |
|--------|------|
| 任务分派集中化 | **有条件支持**——应采用*任务队列*而非有状态的集中调度服务 |
| 沙箱资源集中管理 | **明确反对**——将破坏 gVisor 强隔离边界并引入严重安全风险 |
| LLM API 资源集中管理 | **有条件支持**——仅限多 API Key 场景，以轻量代理实现，无需独立服务 |
| **夜间利用率不足（新增）** | **根本原因是调度问题，而非资源分配问题**——应以常驻 Worker + 自动任务投递解决，与"集中服务层"是不同的方案 |
| 整体"集中服务层" | **不推荐**——是三个不同问题的不当聚合，引入 SPOF 和安全风险 |

## 1. 现状架构分析

### 1.1 当前管道编排模型

参考实现采用**单进程多协程**的扁平编排模型：

```
cli.py (vuln-pipeline)
  └─ asyncio.gather()  ←── N 个并行 run
       ├─ run_0: find_container → grade_container → [judge → report]
       ├─ run_1: find_container → grade_container → [judge → report]
       └─ run_N: ...

共享状态：found_bugs.jsonl（本地文件，仅支持单机）
沙箱：每 run 独立 gVisor 容器，run 结束立即销毁
API：每 Claude CLI 进程直接调用，内置退避重试（最多 20 次）
```

### 1.2 实际性能瓶颈分布

```
单次 run 耗时构成（估计）：

API 响应等待（TTFT/TPOT）  ██████████████████████ ~85%  ← 主瓶颈
ASAN 编译与执行            ████                   ~8%
容器启动（docker run）      ██                    ~2%
编排/调度逻辑               █                    <1%
其他                       ██                    ~4%
```

**关键事实：**
- API 速率限制由 Anthropic **按账号**强制执行，与客户端数量无关
- 容器启动耗时 1–5 秒，相对于数小时的 run，占比可忽略
- `asyncio.gather()` 已在单机上实现 N 路并行，编排开销极低
- Claude CLI 已内置 429 退避重试，pipeline 层还有第二层重试

> **核心结论（修订）：** LLM 资源利用率不足是真实问题，但性质是**时间维度的空闲**（夜间无任务投递，API 配额闲置），而非运行期的 Token 浪费。两者解法根本不同，不可混同。

### 1.3 利用率不足的本质：两种截然不同的问题

经补充信息确认，当前存在的利用率不足属于**类型 A**：

```
类型 A：时间维度空闲（Temporal Idleness）         ← 已确认存在
  现象：夜间 / 非工作时段无任务在运行，API 配额利用率接近 0%
  根本原因：管道依赖人工触发，缺乏自动化调度
  正确解法：常驻 Worker + 自动任务投递（调度问题）

类型 B：运行期 Token 浪费（In-run Inefficiency）  ← 未确认，待度量
  现象：Agent 在等待 ASAN 执行期间 API 空转；单次 run 内并行度不足
  根本原因：提示词设计或工具调用并行化不足
  正确解法：提示词优化 / 并行工具调用（Agent 行为问题）
```

**关键推论：** 解决类型 A 不需要"集中资源分配器"——需要的是一个**持续运行的任务队列**，让 Worker 在没有人工介入的情况下 24 小时消费扫描任务。这与前文 Level 2 方案完全吻合，但优先级应从"按需引入"提升为**优先实施**。

## 2. 集中服务层可行性分析

### 2.1 任务分派集中化

**支持理由：**
- 支持多机水平扩展（单机 asyncio 有 Agent 数量上限）
- 支持跨项目优先级调度（不同目标紧急程度不同）
- 统一的跨机去重状态（取代文件锁，支持多主机并发写入）

**反对理由 / 风险：**
- 集中调度服务本身成为单点故障（SPOF）
- 有状态服务引入分布式一致性问题
- 增加运维负担（服务发现、健康检查、高可用部署）

**裁定：** 有价值，但正确实现是**无状态任务队列**（Redis Streams / SQS），而非有状态的调度服务。

---

### 2.2 沙箱资源集中管理

> ⚠️ **架构风险：明确反对。**

**为什么沙箱不能集中管理：**

**① 安全隔离被破坏**

当前设计的核心安全假设是"每个 run 拥有独立的全新 gVisor 容器"。集中管理容器池将导致 PoC 文件、ASAN 产物、编译残留在容器间泄露，污染后续 run 的结果，甚至为 PoC 注入提供条件。

**② 特权攻击面增大**

集中服务掌控所有沙箱的创建与销毁，一旦被攻破，攻击者可向任意 Agent 注入恶意执行环境。当前设计中每个 Agent 只控制自己的容器，破坏面仅限单次 run。

**③ 收益微小，不值得引入风险**

容器启动 1–5 秒，相对于数小时的 run 耗时，即便将启动时间降至零，对整体吞吐量的改善不足 1%。

```
                ✅ 正确：沙箱本地化                    ❌ 错误：沙箱集中化
──────────────────────────────────────   ───────────────────────────────────────
Worker 自行 docker run gVisor 容器         集中服务维护容器池
  → run 结束立即 docker rm                  → 容器复用给不同 run
  → 每次从镜像快照启动                       → 状态在 run 间泄露
  → 安全边界：Worker ↔ 单容器                → 安全边界：所有容器由单服务控制
  → 攻击面：单次 run                         → 攻击面：所有并发 run
  → 符合 gVisor 设计意图                     → 违反 gVisor 隔离原则
```

---

### 2.3 LLM API 资源集中管理

| 场景 | 可行性 | 推荐方案 |
|------|--------|----------|
| 单 API Key，单团队 | ❌ 集中管理无实质收益 | 现有 Claude CLI 重试逻辑已够用 |
| 多 API Key，多团队共享配额 | ✅ 有价值 | 扩展现有 `egress_proxy.py` 支持多 Key 轮换 |
| 不同优先级任务共享 API 配额 | ✅ 有价值 | Token Bucket 限速器内嵌于代理 |

**结论：** 这里需要的不是"集中服务层"，而是对现有出口代理的轻量增强。

## 3. 优化后的架构设计方案

以**渐进式演进**替代"一步到位的集中服务层"，按实际瓶颈分阶段落地。

### Level 1：轻量扩展（推荐立即评估）

> 适用范围：多机 / 多 API Key / 多团队共享单账号配额

**变更 A：增强 egress_proxy.py → 多 Key 轮换 + Token Bucket**

```
[Agent 0] ──┐
[Agent 1] ──┤──► Enhanced egress_proxy
[Agent N] ──┘     · key_pool = [key_A, key_B, ...]
                  · 429 自动切换下一 Key（round-robin）
                  · 全局 Token Bucket 防突刺
                  · 仅转发 api.anthropic.com:443（白名单不变）

改动文件：scripts/egress_proxy.py（约 50 行新增）
新增依赖：无
```

**变更 B：found_bugs.jsonl → Redis Stream（跨机去重状态）**

```
[Agent 0] ──┐
[Agent 1] ──┤──► Redis XADD/XREAD（append-only stream）
[Agent N] ──┘     · 替代文件锁，支持多主机并发写入
                  · 语义不变（只追加，不修改）
                  · 单机降级：Redis 不可用时 fallback 到文件

改动文件：harness/find.py（dedup 读写逻辑）
新增依赖：redis-py（轻量，无需 broker）
```

---

### Level 2：任务队列 + 常驻 Worker（**优先实施——直接解决夜间利用率问题**）

> 适用范围：夜间 API 配额空闲（已确认）/ 跨物理机调度 / 多目标优先级管理

```
触发源（自动化，无需人工介入）
  ├── cron 定时投递（如每晚 22:00 批量提交扫描任务）
  ├── CI/CD webhook（代码变更触发）
  └── 手动投递（保留兼容）
           │
           ▼
┌──────────────────────────────────────────────────────────┐
│          Coordinator（常驻进程，无状态，可多副本）           │
│  · 维护待扫描目标列表（target + focus_areas + priority）   │
│  · 自动生成 Task = {target, focus_area, model, priority} │
│  · XADD 到 task_stream                                   │
│  · 监控队列深度，防止积压超限                               │
└────────────────────────┬─────────────────────────────────┘
                         │ Redis Streams（持久化，断电不丢任务）
             ┌───────────┴────────────┐
             │                        │
      ┌──────▼──────┐          ┌──────▼──────┐
      │  Worker 0   │    ...   │  Worker N   │
      │ （常驻服务） │          │ （常驻服务） │  ← 24小时运行，夜间持续消费队列
      │             │          │             │
      │ XREADGROUP  │          │ XREADGROUP  │
      │ pull task   │          │ pull task   │
      │     │       │          │     │       │
      │     ▼       │          │     ▼       │
      │ find+grade  │          │ find+grade  │
      │ [gVisor 本地]│          │ [gVisor 本地]│  ← 沙箱始终本地，不集中管理
      │     │       │          │     │       │
      │     ▼       │          │     ▼       │
      │ XADD result │          │ XADD result │
      └─────────────┘          └─────────────┘

夜间利用率改善预期：
  改造前：██████░░░░░░░░░░░░░░░░░░░░░░░░░░░  利用率 ~30%（仅工作时段）
  改造后：████████████████████████████████  利用率 ~85%（队列有任务即消费）
```

**关键设计决策：**

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 任务队列类型 | Redis Streams（XREADGROUP） | 支持消费者组、消息确认、孤儿任务自动重分配 |
| Worker 状态 | 完全无状态 | 可随时崩溃重启，不影响其他 Worker |
| 沙箱分配 | Worker 本地管理 | 安全边界不变，gVisor 隔离不受影响 |
| Judge/Report | 由 Coordinator 消费 result_stream 后触发 | 保持串行化语义，防止 NEW/DUP 竞态 |
| 任务幂等 | Task ID = hash(target + focus_area + run_idx) | 防止 Coordinator 重启导致重复投递 |

## 4. 架构原则对照

| 原则 | 原方案（集中服务层） | 优化方案（队列 + 本地沙箱） |
|------|--------------------|-----------------------------|
| **最小化 SPOF** | ❌ 集中服务即 SPOF | ✅ Worker 无状态，Redis 可 HA |
| **安全隔离** | ❌ 沙箱集中管理破坏 gVisor 隔离 | ✅ 沙箱本地化，边界不变 |
| **最小权限** | ❌ 集中服务掌控所有资源 | ✅ 每个 Worker 只控自己的容器 |
| **渐进式复杂度（YAGNI）** | ❌ 一步引入完整分布式系统 | ✅ 按需分阶段演进 |
| **分离关注点** | ❌ 调度 + 沙箱 + API 混为一谈 | ✅ 三个问题独立解决 |
| **故障隔离** | ❌ 服务故障影响所有 Agent | ✅ 单 Worker 故障不影响其他 |
| **可观测性** | △ 集中日志，但服务本身难调试 | ✅ 每个 Worker 独立 transcript，Redis 提供队列监控 |

## 5. 实施路线图

```
Phase 0（立即，1–2 天）
  补充度量（夜间空闲已确认，仍需量化以指导后续规模）
  · 录制典型工作日 API 使用率曲线（工作时段 vs 夜间对比）
  · 统计 dedup 冲突率（found_bugs.jsonl 文件锁争用频率）
  · 盘点可投入 Worker 的主机数量
  → 输出：目标利用率基线 + 预期 Worker 规模

Phase 1（Level 1，3–5 天）
  前提：需要多 API Key 或跨机去重（单机已足够则可跳过）
  · 增强 egress_proxy.py：多 Key 轮换 + Token Bucket
  · found_bugs.jsonl → Redis Stream（带文件 fallback）
  · 改动范围：~100 行代码，无新服务

Phase 2（Level 2，1–2 周）★ 优先级提升：直接解决夜间空闲
  前提：已确认夜间 API 配额空闲（本次已确认），需自动化运行
  · Redis Streams 任务队列（持久化，支持断电恢复）
  · Worker 改造：从"人工启动"改为"常驻服务 + 拉取模式"
  · Coordinator：cron / webhook 自动投递任务，替代手动触发
  · 孤儿任务处理：XAUTOCLAIM 自动重分配超时任务

Phase 3（可选，4–6 周）
  前提：SaaS / 多租户平台场景，或需要细粒度资源控制
  · 优先级调度（按漏洞严重性 / 目标重要性动态排序）
  · 多租户资源计量计费
  · 可观测性仪表盘（Grafana + Redis 队列深度 + 利用率曲线）
```

## 6. 结语

夜间 API 配额空闲是已确认的真实问题，但其本质是**调度问题**（无任务投递），而非资源分配问题（任务运行时资源争用）。正确的解法是让 Worker 常驻、让任务自动投递，而不是构建一个统管资源的集中服务。

两个核心判断在本次澄清后仍然不变：
- **沙箱不能集中管理**——与利用率问题无关，安全边界约束独立存在
- **集中服务层作为单一组件**——仍是对三个不同问题的不当聚合

推荐的优化方向可以概括为：

> **夜间利用率用常驻队列，任务分派用拉取模式，API 限速用代理，沙箱隔离永远本地。**
