# 补丁生成

管道的 `patch` 阶段接收 `vuln-pipeline run` 结果目录中的已验证崩溃，并生成一个通过可执行验证"梯队"的修复方案。

这是[分类（triage.md）](triage.md)之后的自然步骤：你已有一个经过验证、按优先级排序的崩溃队列，此阶段将每个崩溃转化为一个候选修复，供你审查并提交到上游。

> `/patch` 技能接受静态发现（`TRIAGE.json` 或 `VULN-FINDINGS.json`）或管道运行的结果。对于静态发现（不含 PoC），它运行[运动式补丁生成流程](#运动式补丁生成patch-技能静态模式)。对于管道结果，它委托给 `bin/vp-sandboxed patch` CLI。
>
> 本文主要介绍 CLI，但[审查生成的补丁](#审查生成的补丁)同样适用于静态发现。

> ⚠️ **Patch 评分器会执行目标代码并对其应用模型生成的 diff。** 请与管道其他步骤采用相同的隔离措施。详见 [security.md](security.md)。

> ⚠️ 在将任何变更提交到上游之前，请先阅读下方的[审查生成的补丁](#审查生成的补丁)。本步骤描述的验证梯队尽力验证崩溃已修复，但不保证补丁没有引入新问题。

## 快速上手

Patch 阶段随管道一同提供，无需额外安装。

目标的 `config.yaml` 需要配置 `build_command`，以及可选的 `test_command`，用于验证梯队。本仓库包含的四个目标均已配置。

```bash
# 在管道运行产生 results/<target>/<ts>/ 之后
bin/vp-sandboxed patch results/<target>/<ts>/ --model <model>

# 或在预构建的 canary fixture 上单独试用（无需管道运行）
bin/vp-sandboxed patch targets/canary/fixtures/results_sample --model <model>
```

输出落在 `<results_dir>/reports/bug_NN/{patch.diff, patch_result.json}`，与已有的可利用性报告并列。每次迭代的转录本流式写入 `patch_transcript_itN.jsonl` 和 `reattack_transcript_itN.jsonl`。

## 补丁循环工作原理

Patch 代理在沙箱容器中运行（详见 [agent-sandbox.md](agent-sandbox.md)），获得源代码、PoC、重现命令和 ASAN 跟踪。其提示词要求它修复根本原因而非仅处理崩溃点，查找具有相同漏洞的同级调用点，并尽量保持 diff 最小化。

随后，评分代理在从同一镜像新创建的第二个容器中运行。从第一个容器跨越过来的唯一内容是 diff。评分器看不到 patch 代理的推理过程，因此无法被说服接受一个糟糕的修复。它应用 diff 并爬升下述验证梯队，在任何梯级失败时停止。

若某梯级失败，失败的证据（例如编译器错误、ASAN 跟踪）会注入到下一次尝试的提示词中，补丁循环继续运行，最多进行 `--max-iterations` 次。

## 验证梯队

补丁循环期间，评分代理按顺序运行四项检查，在第一次失败时停止。每项检查都是一个*可执行的 oracle*——运行某些东西并根据结果做出通过/失败决定。没有任何补丁因模型判断而失败。

第五个可选步骤（通过 `--style` 启用）使用模型审查补丁风格，但仅供参考。

| 梯级 | 问题 | Oracle | `patch_result.json` 中的字段 |
|------|------|--------|------------------------------|
| **构建（Build）** | 打了补丁的代码树能编译吗？ | `git apply` + `build_command` 退出码 | `t0_builds` |
| **重现（Reproduce）** | 原始崩溃消失了吗？ | 退出码为 0 且输出中无 `AddressSanitizer:` | `t1_poc_stops` |
| **回归（Regress）** | 破坏了已有行为吗？ | `test_command` 退出码（无测试套件则跳过） | `t2_tests_pass` |
| **重新攻击（Re-attack）** | 根本原因消失了，还是只处理了这个输入？ | 新的 50 轮 find 代理攻击打了补丁的二进制；ASAN 决定结果 | `re_attack_clean` |
| **风格（Style）** | 维护者会接受吗？ | LLM 评判 0-10 分；**仅供参考，从不作为门控** | `t3_style_score` |

当构建、重现、回归（或无测试套件）和重新攻击均通过时，补丁视为通过。

**为什么需要重新攻击？** 能编译并阻止特定 PoC 的补丁通常并不难实现。已发表的模型生成安全补丁评估显示，约 60% 能通过构建和重现检查，但低于 15% 能经受模糊测试和差异测试。主要失败模式是在崩溃点添加边界检查，但让问题值仍然可从略有不同的输入到达。重新攻击步骤防范这种失败模式。

> 将重新攻击通过视为有用的信号，但不是"根本原因已证明修复"。当旁路输入可在 50 轮内构造时，它能很好地区分情况，但对于旁路输入更难构造的错误层次修复，可能会遗漏。

## 审查生成的补丁

验证梯队证明了特定崩溃已修复，但**不能**证明根本原因已被解决，也不能证明 diff 没有引入新问题。梯队不会对 diff 进行语义审查，以检查新漏洞、修复范围外的逻辑变更或其他问题。

你应将 `patch.diff` 视为一个强有力的草稿，在提交到任何地方之前都需要人工阅读。常见问题包括：

- **范围蔓延：** 修改了与崩溃路径无关的文件或函数。
- **抑制而非修复：** `try/except: pass`、对精确 PoC 值提前返回、禁用触发的断言。
- **新攻击面：** 新的解析逻辑、新的从输入信任的大小字段、为使修复"生效"而削弱其他地方的验证。
- **诊断正确，修复错误：** 正确识别了需要修改的模块，但补丁过于局限，破坏了其他内容。

在提交变更前，有两个相对简单的步骤可以采取：

1. 在新会话中运行对抗性验证（*"说出一个能绕过补丁检查并到达同一错误状态的输入变体"*）。Patch 代理已经向自己提出了类似问题，但在新上下文中针对最终的 `patch.diff` 运行更有可能发现修复中的漏洞。
2. 在新会话中要求简化。Patch 代理被提示生成最小化 diff，但其对"最小"的理解受其刚刚推理过的发现所锚定。仅被要求*"简化为修复根本原因的最小变更"*的新上下文传递可以可靠地精简 diff。

> ⚠️ Patch 代理的提示词读取目标衍生数据（ASAN 跟踪、可利用性报告，以及重试时的构建/测试输出）。管道使用每次调用的随机分隔符将这些内容包裹在 `<untrusted_data>` 块中，并指示代理将其视为数据而非指令。但提示级别的围栏是一种缓解措施，不是保证。如果你在运行针对你不完全信任的第三方代码，被污染目标的影响可能在 diff 生成和审查中出现。详见 [security.md](security.md#prompt-injection) 中更广泛的威胁模型。

## CLI 参考

```bash
bin/vp-sandboxed patch <results_dir> --model <m>                   # 为所有唯一漏洞生成补丁
bin/vp-sandboxed patch <results_dir> --bug N                       # 仅为 bug_NN 生成补丁
bin/vp-sandboxed patch <results_dir> --parallel                    # 并发运行 patch 代理
bin/vp-sandboxed patch <results_dir> --no-reattack                 # 跳过梯队中的重新攻击步骤（更快，但较弱）
bin/vp-sandboxed patch <results_dir> --style                       # 运行可选的、仅供参考的风格评判
bin/vp-sandboxed patch <results_dir> --max-iterations N            # 最大补丁循环次数（默认 5）
bin/vp-sandboxed patch <results_dir> --max-turns N                 # 每次迭代的代理预算（默认 200）
```

## 基于 Harness 的重新攻击

重新攻击通常通过在输入文件上运行二进制程序（`./binary <input>`）来驱动目标。对于无法用这种方式驱动的目标（需要启动器、环境设置、多进程编排或非文件输入通道的任何情况），你可以自己提供驱动程序。编写一个了解如何运行目标的脚本，将其放入目标的 Docker 镜像中，并在目标的 `config.yaml` 中通过 `reattack_harness:` 指向它。重新攻击期间，find 代理将候选 PoC 写为 `/poc/` 下的文件并运行你的脚本。补丁循环的其余部分不变。

你的脚本是唯一需要了解目标的东西。代理和管道依赖脚本的退出码。它必须：
- 使用已插桩目标对 `/poc/` 下的每个文件运行（每个 PoC 使用新状态）并捕获每个的 sanitizer 输出
- 如果任何 PoC 使目标崩溃，以退出码 `1` 退出，并打印 sanitizer 跟踪
- 如果每个 PoC 均运行而未崩溃，以退出码 `0` 退出
- 如果目标根本无法启动，以退出码 `2` 退出

## 运动式补丁生成：`/patch` 技能静态模式

`bin/vp-sandboxed patch` 命令依赖 `bin/vp-sandboxed run` 的输出。如果你的发现来自其他来源（独立扫描器、手动审查、仅有文字的报告），或者你在跨多个调用点修复一类漏洞而非一次处理一个崩溃，它将无法工作。

`/patch` 技能的静态模式直接处理这种情况：

```bash
# 为 TRIAGE.json 中严重性最高的 5 个已确认发现起草修复方案
> /patch ./TRIAGE.json --repo ./my-service --top 5
```

对于每个发现，该技能运行两个代理。Patch 代理读取相关代码并将候选修复写成 diff。审查代理随后在干净上下文中评判该 diff，从范围、有效性和引入的新攻击面等角度评估（不执行任何代码）。

不会直接将任何内容应用到你的仓库。生成的 diff 落在 `./PATCHES/` 供你审查，每个结果均标记为 `verified: static_review_only`，以区别于经过验证梯队的结果。

在静态模式下，没有 PoC 可以重新运行，因此回归测试是唯一实际可执行的检查。让 patch 代理在写补丁之前先写一个能重现漏洞的测试。运行它，确认它在当前代码上失败，然后应用修复并确认它通过。没有失败-然后-通过的测试，你无法证明漏洞曾经是真实的，而修复日后可能悄悄退化。

### 当修复实际上是一次迁移时

有时发现的不是一个漏洞，而是一个模式——数十个调用点存在相同的不安全调用，规模太大无法用单个 PR 解决。在这种情况下，你可以运行以下工作流：

1. **研究（Research）。** 代理读取发现和代码库，编写迁移计划：哪个模式不安全、安全的替代方案是什么，以及调用点在哪里。
2. **将计划转化为测试。** 为每个调用点编写一个当前失败、迁移后通过的测试。这个测试套件扮演 ASAN 在管道中的角色，即决定何时完成的检查。
3. **分解为工单。** 将测试分组为可以独立合并的块，每块小到可以审查。
4. **并行修补。** 为每个工单启动一个工作子代理，在其自己的 git worktree 中，循环直到其测试块和项目现有测试都通过。
5. **PR 前门控。** 每个工作者的 diff 必须通过测试、独立的漏洞扫描，以及无工具代理的代码审查（加上你的项目已有的任何检查）。人工审查 PR 本身。
