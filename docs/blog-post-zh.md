# 使用大语言模型保护源代码

模型能力正在快速且不均衡地提升。我们一直与安全团队合作，在他们自己的代码和开源软件中发现并修复漏洞，这项工作让我们对如何利用模型保护源代码有了更深的理解。**我们的主要结论：发现漏洞现在已经可以直接并行化，瓶颈已转移到验证、分类和修补阶段。**

为了说明这种差距，截至 2026 年 5 月 22 日，作为我们自己扫描开源软件工作的一部分，我们已披露了 1,596 个漏洞。据我们所知，其中 97 个已被修补。

本指南介绍如何与 Claude Opus 合作构建威胁模型、发现代码库中的漏洞，然后验证、分类和修补它们。虽然我们没有所有答案，但会分享团队如何扩大发现规模，以及哪些做法在后续阶段有所帮助。*现在就通过[附属仓库](https://github.com/anthropics/defending-code-reference-harness)开始吧*，其中包含交互式工作流的技能和一个自主扫描的演示 harness；我们会在你阅读时指出实现每个步骤的技能。

## 发现-修复循环

发现和修复最多漏洞的团队都在现有最佳实践的基础上形成了某种变体。我们将其提炼为六个步骤的序列：

1. **威胁模型：** 在开始扫描之前，决定什么算作漏洞。
2. **沙箱：** 构建沙箱环境以隔离代理并证明漏洞可利用。
3. **发现：** 让模型在你的源代码中寻找漏洞。
4. **验证：** 独立确认哪些发现实际上可被利用。
5. **分类：** 对发现去重、分配严重性并优先处理需要修复的内容。
6. **修补：** 应用修复，确认漏洞已消除，并搜索变体。

![发现-修复循环概览](../static/defenders-loop.png)

前两个步骤——构建威胁模型和沙箱——是后续循环的设置工作。这些通常每个代码库只做一次，并在底层系统变化时重新审视。后四个步骤是你针对源码运行的循环：发现、验证、分类和修补。

对代码库的第一次运行通常发现最多问题。后续运行往往发现更少——尽管通常更复杂——的漏洞，因为更简单的已在之前的运行中被修补。但是，不要期望第 n 次运行零新发现。模型是随机性的，大型代码库可能有长尾漏洞，即使代码不变，也会持续涌现。

在首次迭代某个代码库时，你应该多次运行循环，根据净新发现的数量和该系统的风险承受能力来决定何时停止。在那之后，继续（1）定期或（2）每当代码发生有意义的变化时扫描。

接下来，我们将详细介绍每个步骤，解释为什么重要、产出什么，以及如何实施。

## 1. 威胁模型：定义什么算作漏洞

假阳性最常见的原因是模型对你的信任边界缺乏良好理解。模型可能将代码标记为有漏洞，因为它假设客户端可以发送损坏的值或攻击者可以控制配置，即使这些输入在你的环境中是*可信的*。反之，模型可能假设一个面向互联网的服务是内部服务，从而漏报真实漏洞。两种情况下，模型对威胁模型而非代码的判断是错误的。

*一个团队注意到发现中的一个模式：当系统有良好记录的威胁模型、系统设计文档、需求和约束时，模型表现最好。威胁模型定义良好时，模型的发现 90% 都是可利用的。*

你可以通过两个步骤与 Claude 合作构建威胁模型：

**首先，从代码、文档和漏洞历史中引导。** 给模型提供你会在第一天交给新安全工程师的内容：架构文档、wiki、入口点、git 历史和过去的漏洞。这有助于克服仅从代码推断隐式知识、权衡和设计决策的挑战。然后，要求模型创建一个包含系统上下文、资产、入口点和信任边界的威胁模型。最后，让模型对过去的漏洞进行聚类，并列出相关漏洞类别。确保威胁模型记录了你关心和不关心哪些漏洞及原因。

*一个团队审查了数百个过去的 CVE 和安全修复提交，将其提炼为"漏洞形态"提示，并向模型提出两个问题：修复是否完整，以及它是否在其他地方也得到了应用？他们在一小时内发现了三个可利用的问题。正如他们所说："'人们过去利用了什么'有时是比'在这个代码库中找到漏洞'更简单的作弊码。"*

**其次，让模型采访一个了解系统的人。** 考虑 [Shostack 的四个问题](https://github.com/adamshostack/4QuestionFrame)：*我们在构建什么？什么可能出错？我们在对此做什么？我们做得好吗？* 先运行引导步骤，这样受访者不必从零开始。这样，他们可以从草稿出发，而不是花几个小时从头研究和构建威胁模型。虽然访谈步骤是可选的，但它添加了模型无法从代码或文档中获得的上下文，从而改进威胁模型。

一些实践可以带来很大不同：

* **考虑你的依赖项的安全策略。** 许多开源项目会发布安全策略。例如，vLLM 的 [`security.md`](https://docs.vllm.ai/en/latest/usage/security.html)、SQLite 的["防御黑暗艺术"](https://www.sqlite.org/security.html)，以及 [ImageMagick 的安全策略](https://github.com/ImageMagick/ImageMagick/security/policy)。你的威胁模型应直接考虑这些，而不是从头重建策略。
* **明确什么是*可信的*。** 如果你信任配置文件或已认证客户端，在威胁模型中记录这一点。这些假设有助于区分不可利用的漏洞与真实漏洞。
* **在代码中包含 `THREAT_MODEL.md`。** 将其放在仓库中，并随代码变化更新。发现代理可以在搜索前读取它，跳过已知的非问题。

威胁模型在两个地方会用到。在发现中作为范围：对代码进行分区、优先处理目标，并跳过超出范围的内容——这对无法完全扫描的大型代码库很有帮助。在分类中作为过滤器：广泛扫描后，使用威胁模型更好地根据你的系统和环境校准严重性。

*一个扫描大型项目的团队有 40% 的假阳性率，并深入调查原因。发现是可重现的，PoC 证明了可利用性。但拥有代码的开发团队将其视为假阳性，因为这些漏洞不符合项目的威胁模型。另一个团队的 CISO 一针见血地指出："[模型]对代码有很好的上下文，但对我们没有很好的上下文。"*

**尝试 [`threat-model` 技能](https://github.com/anthropics/defending-code-reference-harness/tree/main/.claude/skills/threat-model)。** 它走过本节描述的两个步骤——`bootstrap` 从你的代码、CVE 和 git 历史推导草稿，`interview` 引导系统负责人回答 Shostack 的四个问题以完善它。输出是一个 `THREAT_MODEL.md` 文件，用于发现和分类步骤。

## 2. 沙箱：安全运行代理并验证可利用性

**沙箱的一个目的是保护你的系统。** 要使模型能够安全自主地运行，你需要一个强隔离层。没有它，代理可能超出目标范围并做出意外的事情。

*一个团队告诉模型它没有网络访问权限——实际上有——模型发现它可以从 GitHub 获取数据。另一个团队观察到一个代理在扫描过程中回答了一个 GitHub issue。这两个操作都不是恶意的，但都证明了需要通过代码和配置来强制执行约束。*

将隔离与你的威胁模型匹配。容器适用于读取代码的发现代理，但在 microVM（如 Firecracker）或完整 VM 中运行目标和其 PoC，并锁定出口，以防止任何内容到达你的生产系统。绝不要让代理可以访问凭据（`~/.aws`、`~/.ssh`、`.env`）。

只在设置沙箱时给它网络访问权限。拉取依赖、构建、安装工具、部署目标，并运行现有测试以确认一切正常。然后，对环境进行快照并移除其网络访问。扫描期间，只允许流量到达模型 API，通过本地代理路由。在每次运行开始时加载快照，使每次扫描从相同的干净起点开始。

**沙箱的另一个目的是证明可利用性。** 在静态扫描期间，模型读取代码并假设什么可能中断，但无法测试路径是否可达或是否有补偿控制。因此，模型可能标记你实际上不关心的、不可利用的代码正确性漏洞。当团队构建了一个代理可以编译代码、运行测试和引爆 PoC 的沙箱时，不可利用的发现显著减少。

*一个进攻性安全团队构建了一个给代理提供测试床的 harness，带有一个简单的验证规则：只有当代理能够构建 PoC 并在测试床上运行时，才是真正的正例。他们六周后的评估是："最大的效能杠杆是给模型测试床、实时系统和运行 PoC。"*

构建沙箱时，尽量固定所有内容，使每次运行在相同环境中使用相同代码：镜像标签、commit SHA、依赖和构建命令。缓存本地副本以便构建无需网络，并力求使容器持久，以便多个测试循环可以直接加载它。

*一个团队的扫描标记了一个漏洞，结果发现这是代理下载了旧版本的库而非实际部署版本的副作用。这被一位读取转录本并发现下载了不同依赖项的工程师发现了。他们现在构建 Docker 容器，将依赖固定为与生产版本匹配，使发现代理和验证代理操作与攻击者实际面对的相同工件。*

构建足够忠实于生产环境的沙箱非常重要。排除依赖项（如队列或数据存储）可能导致漏报生产中可能存在的漏洞。反之，忽略生产防御（如 WAF 或认证网关）会导致模型报告你的生产环境已缓解的不可利用发现。

尽管如此，如果由于云依赖、数据存储或其他现实世界复杂性，构建有代表性的沙箱不切实际，可以先从下面的发现步骤开始。你不一定需要在沙箱中运行 PoC。前沿模型擅长仅通过分析源代码发现漏洞。几个团队，包括我们自己，已发现这很有效。权衡在于验证阶段——没有运行目标，我们无法用 PoC 证明发现，因此需要为验证投入更多时间。你也可以稍后在发现量证明其合理时投资沙箱。

**参阅 [harness README.md](https://github.com/anthropics/defending-code-reference-harness/tree/main/harness) 获取参考沙箱。** 在此实现中，代理和目标在 gVisor 隔离的容器中运行，出口锁定到模型 API。目标从固定到特定提交的 Dockerfile 构建，[`setup_sandbox.sh`](https://github.com/anthropics/defending-code-reference-harness/blob/main/scripts/setup_sandbox.sh) 处理设置阶段。

## 3. 发现：提供丰富上下文、较短提示和有用工具

给发现代理访问它可以按需加载的上下文，例如威胁模型、架构文档和过去扫描的结果。当代理理解你的信任边界和系统的实际部署方式时，它能更好地识别特定于你的系统的漏洞。

我们发现前沿模型在发现阶段受益于越来越简单的提示。反直觉地，更规定性的提示会使发现变得更差——长篇检查清单往往会降低模型的创造力并产生更少的新颖漏洞。以下是在发现阶段有帮助的一些提示技巧：

* **提供目标和上下文。** 指出"为什么"和"是什么"——为什么扫描、什么样的发现是重要的、扫描的是什么系统——并将"如何扫描漏洞"留给模型。前沿模型在安全任务上越来越好，过于规定性可能会缩小它们尝试的范围。
* **尝试要求特定漏洞类别。** 如果你想专注于由先前 CVE 或代码库语言引导的特定漏洞类型，请说明。描述漏洞类别、它的作用和倾向于存在的位置，以便模型能在你的代码库中识别它。
* **定义输出。** 要求结构化报告包含预定义字段，并按顺序排列，使模型的推理建立在每个字段上。示例字段包括理由、发现、影响、严重性等。包含一个逃生舱口，让模型可以提前退出以处理弱发现。

给模型提供搜索和阅读代码库的工具，如 grep、glob 等。也让模型使用你的团队可能使用的安全特定工具，如 SAST 扫描器或模糊测试器。询问模型特定任务需要什么工具并提供它们。最后，让模型按需构建工具：最近的前沿模型越来越擅长编写它们需要的工具。

*除了源代码，一个渗透测试团队还给发现代理提供了发送请求、检查响应和查询流量日志的工具。因此，代理不需要猜测路径是否可达，可以随着工作对运行中的应用程序测试每个候选，将其真正正例率提高到接近 100%。*

让模型对系统进行第一次传递，对搜索空间进行分区，例如按攻击面、端点或组件。然后，将这些分区提供给并行发现代理，使它们不会收敛到相同的浅层漏洞。最后，运行系统级传递，将分区级发现作为上下文来搜索漏洞。

*尝试暴力扩大发现规模的团队很快遭遇边际收益递减。来自一个团队："我们最初尝试只是水平扩展并发送更多代理，但看到了限制性回报。"另一个增加了焦点区域数量和并行代理数量，得到了"大量问题"，大多数相互重复。*

如果你有一个沙箱可以运行目标，要求发现代理构建发现的 PoC，例如脚本、崩溃输入或失败测试。构建 PoC 有助于代理迭代和确定发现，而工件给验证代理提供了具体证据进行评估。尽管如此，代理无法重现的发现仍然可以被报告，标记为未经证实，这样你可以保持高召回率。

**[`vuln-scan` 技能](https://github.com/anthropics/defending-code-reference-harness/tree/main/.claude/skills/vuln-scan)** 在此阶段很有帮助。它读取你的 `THREAT_MODEL.md`，将目标分区为焦点区域，并为每个区域展开并行审查代理。输出是后续步骤直接使用的结构化发现。

## 4. 验证：过滤不可利用的发现

发现优化召回率；验证优化*精准率*。换句话说，发现应找到尽可能多的漏洞——甚至是不太可能的——而验证应排除实际上不可利用的发现。当代理试图在同一步骤中*同时*执行这两项时，它可能自我审查并排除可利用的真正正例。我们以惨痛代价学到了这一点：要求发现代理同时验证发现，导致它们过滤掉了独立验证步骤本应确认的真正正例。

验证器代理应独立于发现代理。在没有共享文件系统或对话历史的全新容器中运行验证器。如果验证器接触到发现代理的推理，它可能只是同意而非测试声明。因此，只给验证器（1）PoC 或书面发现和（2）代码库，以便它可以搜索发现者遗漏的缓解措施（例如，上游验证、认证门、类型约束或不可达代码）。

如果单次验证传递仍然让太多不可利用的发现通过，尝试运行多个独立验证器。它们可以考虑不同角度或使用不同模型运行。然后取多数票。也考虑有一个单独的 judge 来决定发现和验证代理结果之间的分歧。

提示验证代理反驳发现代理的发现。让验证器假设每个发现是假阳性，并搜索发现错误的原因。包括验证器代理可以用来确定发现是否为真正正例的明确标准。这在发现代理的输出不包含 PoC 时最为重要。尽量排除尽可能多的不可利用发现，以减少手动审查的工作量。

*在我们合作过的团队中，添加对抗性验证器大约将发现阶段不可利用发现的比例减半。要求该验证器还构建证实漏洞利用的 PoC 将假阳性率降至接近零。这两个步骤合在一起，有助于显著减少下游分类和修补的工作量。*

如果你能够在沙箱中充分重现生产环境（见步骤 2），提示验证器代理构建并执行可重现的 PoC。如果 PoC 有效，你可以得出发现是可利用的结论。注意，反过来不成立——无法生成有效 PoC 并不能证明是假阳性。

*一个扫描开源包的团队构建了一个有助于闭合循环的验证步骤：扫描包，生成 PoC，然后部署一个使用该包并触发 PoC 的模拟应用程序。他们的观点是："验证是最大的障碍，PoC 是验证手段。"*

## 5. 分类：按根本原因去重，按前提条件和影响排序

虽然验证确认发现可利用，分类评估修补优先级。以前，当发现需要更多努力时，发现漏洞的工程师也会对其进行分类。现在，随着模型能够在午饭前找到一百个候选，分类已成为瓶颈。

适当的分类有助于防止告警疲劳。如果你提交了太多重复或严重性虚高的漏洞，产品工程师可能会停止阅读，即使是那些需要立即修补的。开源维护者尤其可能被未经分类的发现所淹没，因为他们从许多依赖其软件的不同用户那里收到报告。

*多个团队分享了同样的教训：如果我们向产品工程师发送一堆大多数不可利用的发现，他们会失去对报告的信任并放弃。他们还优先处理严重和高危发现，以避免让下游工程师不知所措。其他团队发现了一个成功之道：将模型指向他们现有的积压——来自先前扫描器、先前模型、漏洞赏金收件箱的未解决发现——在几天内清理了数百个过期项目。*

要对发现去重，考虑根本原因。扫描器经常在多个调用点标记同一个漏洞，或报告单个根本原因的多个症状。以下是一个实用方法：首先，使用廉价的确定性传递：同一文件、同一类别、漏洞行号在十行以内。然后，让模型对剩余内容应用定性规则：

* 视为重复：措辞不同的相同根本原因；在多个调用点报告的相同漏洞；报告每个端点缺少的全局保护（如认证检查）；在同一路径中标记的原因和其后果。
* 视为不同：同一文件中不同的漏洞类别；到达不同汇点的不同变量；一个辅助函数中的两个独立漏洞；两个端点上缺少相同检查，但每个都需要自己的修复。

如果你的 harness 为每个发现生成 PoC 和补丁，去重发现的另一个方法是检查一个发现的补丁是否也能解除其他发现的 PoC。

去重后，根据以下标准评估每个发现的严重性：

* **可达性。** 攻击者能从真实入口点到达此代码，还是只能从内部代码和端点到达？
* **攻击者控制。** 不可信输入是否完整到达汇点，或者上游有什么内容对其进行净化或约束？
* **前提条件。** 触发漏洞需要什么条件：非默认设置、特定功能标志、攻击者必须命中的窄时间窗口？
* **认证。** 未认证的攻击者能触发它，还是需要登录用户或管理员？
* **读取 vs. 写入。** 攻击者只能读取数据，还是也能修改它？
* **影响范围。** 如果 PoC 触发，谁会受影响？一个用户还是所有用户，一个租户还是整个平台，用户空间还是内核？

要将评估标准转化为评分，让模型在分配严重性之前写出每个问题的答案。先梳理证据可以防止模型根据漏洞类别（"SQL 注入，所以严重"）固定锚点，然后虚高严重性以匹配。作为起点：零前提条件且具有未认证远程访问为严重或高危；一两个前提条件或认证路径为中危；三个或更多，或仅本地为低危。根据你的系统调整阈值。

模型可能会夸大严重性，因为它们上下文不足。它们可能不知道攻击者实际控制哪些输入，或者看不到补偿控制。作为前者的例子，如果 SQL 注入由未认证请求触发则为严重，但如果由仅管理员配置文件触发则不是问题。对于后者，防止利用的上游 WAF 或认证可能从源代码单独来看不可见。

解决方案是在分类期间提供一个威胁模型，告诉模型在你的系统中哪些类型的漏洞你关心，哪些不关心。例如，澄清"我们信任已认证客户端"可以简化或消除整类严重漏洞。

*一个团队发现模型经常过于自信，除非有某些东西可以验证，或者对于威胁模型中预期的某些内容有更多上下文。他们的解决方案是给分类代理提供与发现代理获得的相同威胁模型。*

**尝试 [`triage` 技能](https://github.com/anthropics/defending-code-reference-harness/tree/main/.claude/skills/triage)。** 它同时执行验证和分类：每个发现的多票验证、跨运行去重，以及按推导的可利用性重新排序。输出是一个简短的、按优先级排序的、已分配责任人的列表，而不是原始转储。

## 6. 修补：闭合循环并改善下一轮的上下文

修补是你闭合循环并修复漏洞的地方。它还有助于根据验证的发现改进威胁模型——更新信任边界或需要更多审查的组件——并将过去的发现输入到下一次扫描的上下文中。每个循环都会加固代码库，使下一次扫描更有针对性。

在修补前，编写一个在现有代码上失败的新测试。然后实施修复，确认同一测试现在通过，且没有破坏其他任何内容。（是的，这是测试驱动开发。）如果不添加测试，修复可能悄悄退化，而且很难事后证明漏洞是真实的。

*一个渗透测试员发现他们生成的补丁不一致——有些好，有些差——直到 harness 告诉模型通过对已打补丁的代码重新运行 PoC 来验证补丁。通过给模型提供迭代的反馈，补丁质量提升，节省了人工审查时间。*

模型可能会在特定调用点而非根本原因上处理发现。简单地提示模型识别并修复根本原因可能很有效。然后，让模型在两个层次上寻找变体：（1）相同模式，其他地方是否有相同有漏洞代码的其他调用点或副本；（2）相同类别，具有一个 SQL 注入漏洞的代码库倾向于有更多 SQL 注入漏洞。用验证的发现和补丁更新威胁模型以闭合循环。

发布补丁前，运行对抗性检查。让新的发现代理作为攻击者探测补丁，确认补丁是全面的。然后，简化生成的补丁以处理过于侵入性的补丁。最小化补丁更容易审查，更不可能引入新漏洞。提示生成修复根本原因的最小变更——不重构、不顺手清理、不重新格式化。

*一个团队关于他们最常见的补丁失败："推荐的补丁往往尽可能地限制性，以至于会破坏与其他服务的连接。它会解决问题，但会破坏允许服务工作的依赖关系。"*

你可以通过一个检查梯队验证每个补丁，从最便宜的开始：

1. **构建。** 补丁编译且新测试通过。
2. **尝试重现。** 原始 PoC 应该停止工作。这会捕获无效补丁。
3. **检查回归。** 原始测试套件仍然通过。这会捕获破损或过于限制的补丁。
4. **重新攻击。** 一个新的发现代理运行对抗性检查。这会捕获不完整的补丁。

最后，虽然模型可以编写补丁，但人类仍然需要拥有它。生成的补丁可能以可预测的方式失败——修复症状而非根本原因、阻止合法输入或移除对依赖服务的访问。目标是尽可能多地验证每个补丁，使人工审查需要更少的工作量。目标是帮助开发团队专注于模型可能不知道的细微差别（例如，即将到来的变更、代码风格），补丁需要最少的审查和更新。

**尝试 [`patch` 技能](https://github.com/anthropics/defending-code-reference-harness/tree/main/.claude/skills/patch)。** 它使用分类输出，为每个发现生成一个候选 diff，并有独立的审查代理检查每个。

## 开始上手

亲自尝试运行该循环。克隆 [`defending-code-reference-harness`](https://github.com/anthropics/defending-code-reference-harness) 并在 Claude Code 中运行 `/quickstart`。它带你通过一个交互式工作流，从威胁建模到扫描再到分类，针对一个演示目标。该仓库还包含一个自主 harness 和一个 `/customize` 技能，用于为你的环境更新 harness。

然后，在你自己的代码上运行它。选择一个服务或包。从代码和文档引导威胁模型，并进行访谈。投资构建你的环境沙箱。扫描。用独立代理验证发现。根据你的标准进行分类，并审查所有评级为高危及以上的内容。修补。然后定期重新扫描。

你的第一次扫描将发现比预期更多的问题。大多数需要验证和分类。在为更多扫描预算之前，先为扫描*之后*的管道预算。

## 展望未来

我们相信模型[发现和利用代码漏洞](https://red.anthropic.com/2026/exploit-evals)正在变得越来越容易。因此，我们作为防御者的工作是在对手利用之前在我们的代码中发现并修复漏洞。一些团队甚至将他们的 harness 连接到事件：漏洞赏金报告触发自动变体分析，安全审查触发扫描并附加候选发现，验证的漏洞更新静态分析工具以防止未来出现。

这项工作至关重要且风险极高。但做对了，这是一个更大、更充满希望的转变的开始，在那里我们将*能够*在攻击者利用之前发现并修复漏洞。
