# 内部漏洞挖掘平台 — 整体架构设计文档

> 本文档为面向多产品研发团队的内部漏洞挖掘平台架构设计，采用八层架构，覆盖从代码接入、自动化分析、AI 驱动漏洞挖掘、自动修复与验证，到研发团队消费报告的完整闭环。

# 1. 文档概述

## 1.1 背景与目标

随着公司业务规模持续扩张，各产品线代码量与依赖复杂度显著增加，依赖人工代码审计与周期性渗透测试的传统安全模式已难以满足需求。本平台覆盖从代码接入到研发消费的完整漏洞处理闭环。

核心目标：

- 将漏洞发现周期从"按需渗透测试"压缩为"持续自动化扫描"
- 通过 AI Agent 协作，将高质量漏洞报告（含 PoC、根因分析、修复建议）直接推送给研发团队
- 引入 AI 自动修复与验证闭环，大幅降低修复成本，缩短漏洞修复 SLA
- 为所有产品线提供统一的安全基线与可观测性

## 1.2 适用范围

| 范围 | 说明 |
|------|------|
| 产品类型 | Web 应用、移动端、后端服务、SDK、内部工具 |
| 语言支持 | C/C++、Java、Python、Go、JavaScript/TypeScript |
| 目标环境 | 应用层代码、OS 内核驱动（特定产品） |
| 团队角色 | 安全团队（平台运营）、各产品研发团队（消费端） |

## 1.3 术语表

| 术语 | 含义 |
|------|------|
| PoC | Proof of Concept，漏洞概念验证代码，能够触发漏洞的最小可复现输入 |
| AST | Abstract Syntax Tree，抽象语法树，代码的树形结构化中间表示 |
| CVSS | Common Vulnerability Scoring System，通用漏洞评分系统 |
| CWE | Common Weakness Enumeration，通用缺陷枚举 |
| Agent | 由 LLM 驱动、能自主调用工具并迭代推理的自动化任务执行单元 |
| Fuzzing | 模糊测试，通过大量随机/变异输入发现程序异常行为 |
| diff | Git 标准补丁格式，描述代码变更内容 |
| RAG | Retrieval-Augmented Generation，检索增强生成 |

# 2. 整体架构概览

## 2.1 架构分层（八层）

平台采用八层架构，其中第 ⑤⑥ 层为新增的自动修复与验证模块。

```
┌─────────────────────────────────────────────────────────────┐
│ ① 接入层    Git Hook · CI/CD · API 提交 · 定时扫描            │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌──────────────────┬──────────────────┬─────────────────────────┐
│ ② 静态分析引擎   │ ② 动态分析引擎   │ ② 知识库 / RAG          │
│ CodeQL/Semgrep   │ Syzkaller/AFL++  │ CVE · CWE · 历史漏洞    │
└──────────────────┴──────────────────┴─────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ ③ Agent 编排层（Planner）任务拆解 · 调度 · 并行/串行控制      │
│   ├─ Researcher Agent  相似漏洞检索                            │
│   ├─ Analyst Agent     代码深度分析                            │
│   ├─ Exploiter Agent   PoC 构造与验证                          │
│   └─ Patcher Agent     修复方案生成                            │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ ④ 沙箱漏洞验证层  Docker 隔离 · KASAN · PoC 验证 · CVSS 评分  │
└─────────────────────────────────────────────────────────────┘
                            ↓ [验证通过]
┌─────────────────────────────────────────────────────────────┐
│ ⑤ 自动修复引擎 [NEW]                                          │
│   补丁生成 · 多方案排序 · 人工审核门控                        │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ ⑥ 修复验证闭环 [NEW]                                          │
│   回归测试 · 副作用检测 · 补丁安全扫描                        │
│   [验证失败] → 重试（最多3次）→ 超限转人工                    │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌──────────────────────────────┬──────────────────────────────┐
│ ⑦ 漏洞管理 去重·优先级·状态机│ ⑦ 报告与通知 PR·Jira·Slack   │
└──────────────────────────────┴──────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ ⑧ 研发消费端  审核 · 合并 · 确认                              │
└─────────────────────────────────────────────────────────────┘

底座: LLM 核心（Claude API / 私有部署）驱动全部 Agent 推理
回路: 修复合并后 → 重新触发全量扫描确认根因消除
```

各层职责详表：

| 层次 | 名称 | 核心职责 |
|------|------|----------|
| ① 接入层 | 代码与触发接入 | Git Hook、CI/CD、API 手动提交、定时全量扫描 |
| ② 分析引擎层 | 静态 + 动态 + 知识库 | CodeQL/Semgrep 静态分析；Syzkaller/AFL++ 动态 Fuzzing；CVE/历史漏洞 RAG |
| ③ Agent 编排层 | 多 Agent 协作 | Planner 调度 Researcher、Analyst、Exploiter、Patcher 四类专职 Agent |
| ④ 漏洞验证层 | 沙箱执行与评分 | Docker 隔离执行 PoC，KASAN/sanitizer 检测，CVSS 自动评分 |
| ⑤ 自动修复引擎 | 补丁生成（新增） | LLM 生成多候选补丁，按安全性/侵入性/置信度排序，人工审核门控 |
| ⑥ 修复验证闭环 | 补丁有效性确认（新增） | 回归测试、副作用检测、补丁安全扫描，失败自动重试 |
| ⑦ 漏洞管理层 | 生命周期管理 | 去重、优先级排序、状态机跟踪、Jira/Slack 集成 |
| ⑧ 研发消费端 | 报告与合并入口 | Web 看板、IDE 插件、PR 一键合并、邮件通知 |

## 2.2 数据流向

### 正向主流程（发现 → 修复）

```
代码变更 → 静态+动态分析 → Agent协作 → 沙箱验证 →[通过]→ 自动修复 → 修复验证 →[通过]→ PR创建 → 研发合并 → 确认扫描
```

1. 代码变更通过接入层触发扫描任务
2. 静态引擎构建 AST/数据流图，动态引擎生成 Fuzzing 语料，知识库提供历史参考
3. Planner Agent 接收分析结果，并行调度四类子 Agent 协作执行
4. Exploiter Agent 构造 PoC，沙箱验证层确认漏洞可触发，计算 CVSS 分数
5. 验证通过的漏洞进入自动修复引擎，生成多个候选 diff 补丁
6. 修复验证闭环对补丁执行回归测试、副作用检测、补丁安全扫描
7. 通过验证的补丁自动创建 PR，推送至对应研发团队，附完整漏洞报告
8. 研发审核后合并，修复结果写回平台触发最终确认扫描

### 反馈回路（修复验证失败）

- 补丁验证失败 → 自动重试，最多 3 次重新生成补丁
- 超过重试上限 → 标记为"需人工处理"，转安全团队介入
- 修复合并后 → 重新触发全量扫描，确认漏洞根因彻底消除

# 3. 各层详细设计

## 3.1 接入层

### 3.1.1 设计原则

核心目标是**零摩擦集成**——研发团队无需改变日常工作流程，平台自动感知代码变更并触发扫描。

### 3.1.2 接入方式

| 接入方式 | 触发时机 | 适用场景 | 优先级 |
|----------|----------|----------|--------|
| Git Hook（pre-push） | 代码推送前 | 增量变更快速扫描 | 高 |
| Merge Request 触发 | PR 创建/更新时 | 合并前门控检查 | 高 |
| CI/CD Pipeline 集成 | 构建流水线中 | 与现有 Jenkins/GitLab CI 集成 | 高 |
| API 手动提交 | 按需触发 | 安全团队专项审计 | 中 |
| 定时全量扫描 | 每周/每月 | 覆盖历史遗留代码 | 低 |

### 3.1.3 代码预处理

- 构建 AST（抽象语法树）：解析源码结构，去除注释、格式等与语义无关的细节
- 提取函数调用图（Call Graph）：识别跨文件的函数依赖关系
- 生成代码摘要：按模块/文件级别生成结构化摘要，供 LLM 消费
- 变更差异计算：增量模式下，仅提取相对上次扫描的变更部分

> **AST 说明**：AST（抽象语法树）是编译器将源代码转换成的树形数据结构，保留代码的结构和语义，去除空格、括号等格式噪音。例如 `x = 1 + 2 * 3` 对应一棵以"赋值"为根节点的树，乘法作为加法的子节点，天然编码了运算优先级。静态分析工具在 AST 之上进行数据流追踪和模式匹配。

## 3.2 分析引擎层

### 3.2.1 静态分析引擎

| 工具 | 核心能力 | 适用场景 |
|------|----------|----------|
| CodeQL | 语义级数据流/污点追踪，QL 查询语言 | 跨函数注入漏洞、权限检查缺失 |
| Semgrep | 基于规则的快速 Pattern 匹配 | 已知漏洞模式快速扫描 |
| Joern | 代码属性图（CPG），跨语言分析 | C/C++ 内存安全类漏洞 |
| Tree-sitter | 轻量 AST 解析 | 为 LLM 生成结构化代码摘要 |

### 3.2.2 动态分析引擎

| 工具 | 核心能力 | 适用目标 |
|------|----------|----------|
| Syzkaller | 系统调用 Fuzzing，覆盖率引导，KASAN 集成 | OS 内核、驱动程序 |
| AFL++ | 覆盖率引导 Fuzzing，结构感知变异 | C/C++ 应用层二进制 |
| LibFuzzer | 进程内 Fuzzing，LLVM 集成 | 库函数级 Fuzzing |
| Frida | 运行时动态插桩，内存分析 | 黑盒应用行为分析 |

### 3.2.3 知识库与 RAG

- CVE 数据库：NVD 同步，包含漏洞描述、CVSS 评分、修复方案
- CWE 分类库：通用缺陷枚举，用于漏洞类型标注
- 公司内部历史漏洞库：历次发现的漏洞及对应修复方案
- 相似漏洞补丁库：历史修复 diff 的向量化存储，供 Patcher Agent 检索参考

## 3.3 Agent 编排层

### 3.3.1 Planner Agent 职责

- 接收来自分析引擎的结构化结果，分解为子任务
- 根据漏洞类型、代码语言、项目上下文，决定调用哪些子 Agent 及调用顺序
- 管理并行与串行执行：多产品线任务并行处理，单漏洞分析流水线串行执行
- 汇总各子 Agent 输出，构建最终漏洞报告

### 3.3.2 四类专职子 Agent

| Agent | 输入 | 核心工作 | 输出 |
|-------|------|----------|------|
| Researcher | 代码摘要、静态分析结果 | 检索相似 CVE 和历史漏洞，提供攻击模式参考 | 相似漏洞列表、攻击向量参考 |
| Analyst | 可疑代码片段、调用图 | 深度分析漏洞根因，追踪数据流路径，评估可触发性 | 根因分析报告、可达路径描述 |
| Exploiter | 根因分析结果 | 构造能触发漏洞的 PoC 输入，迭代验证触发条件 | 可执行 PoC 代码、触发步骤 |
| Patcher | 漏洞详情、PoC、历史修复参考 | 生成多个候选修复补丁，分析修复方案权衡 | 多个 diff 补丁 + 修复说明 |

### 3.3.3 LLM 核心选型

- **云端方案**：Claude API（Sonnet 4.6 / Opus 4.8），代码理解与推理能力强
- **私有部署方案**：开源模型（Qwen-Coder、DeepSeek-Coder）私有化部署，适合对代码安全性要求高的场景
- **混合方案**：普通漏洞走云端 API，涉及核心代码的高危漏洞走私有部署模型

> ⚠ **安全提示**：代码库可能包含核心业务逻辑和商业秘密。使用云端 LLM API 时，应对代码进行脱敏处理（变量名混淆、字符串替换），或采用私有部署方案，避免将原始代码发送至第三方服务。

## 3.4 沙箱漏洞验证层

### 验证流程

1. 在独立 Docker 容器中构建目标项目（开启 ASAN/KASAN/UBSAN）
2. 执行 PoC，捕获 crash 信息（崩溃地址、栈帧、内存状态）
3. 分析 crash 类型（堆溢出、UAF、空指针等），映射到 CWE 分类
4. 评估漏洞影响范围，计算 CVSS v3.1 评分
5. 生成标准化漏洞记录，写入漏洞管理数据库

### 容器隔离规格

| 配置项 | 规格 | 说明 |
|--------|------|------|
| 容器运行时 | Docker + gVisor | gVisor 提供额外的内核隔离层 |
| 网络策略 | 完全隔离 | 禁止容器访问外网和内网 |
| CPU/内存限制 | 2 核 / 4GB | 防止 DoS 类 PoC 耗尽宿主资源 |
| 执行超时 | 60 秒 | 超时强制终止，视为 PoC 无效 |
| 文件系统 | 只读挂载 + tmpfs | 防止 PoC 修改宿主文件 |

## 3.5 自动修复引擎（新增）

### 3.5.1 补丁生成策略

- 每次生成 3~5 个候选方案，覆盖不同修复思路（输入校验、边界检查、安全 API 替换）
- 每个方案输出标准 git diff 格式，可直接应用于代码库
- LLM 同时生成修复说明文档，解释漏洞根因、修复逻辑和潜在风险

### 3.5.2 候选方案评分维度

| 评分维度 | 含义 | 评估方法 |
|----------|------|----------|
| 安全性 | 是否彻底消除漏洞根因 | 静态分析重新扫描补丁后代码 |
| 侵入性 | 对现有代码改动的范围 | 计算 diff 行数、影响函数数量 |
| 置信度 | LLM 对修复方案正确性的把握 | 参考相似 CVE 修复案例的匹配度 |
| 兼容性 | 是否可能破坏现有接口 | 分析公开 API 签名变化 |

### 3.5.3 人工审核门控（按 CVSS 分级）

| CVSS 分数 | 风险等级 | 处理策略 |
|-----------|----------|----------|
| 0.1 – 3.9 | 低危 | 全自动提交 PR，无需人工审核，72 小时内合并 |
| 4.0 – 6.9 | 中危 | 自动生成 PR，需研发 Lead 审核后合并，48 小时 SLA |
| 7.0 – 8.9 | 高危 | 生成补丁草稿，强制安全团队 + 研发 Lead 双重审核 |
| 9.0 – 10.0 | 严重 | 立即告警安全团队，人工制定修复方案，24 小时紧急响应 |

## 3.6 修复验证闭环（新增）

### 3.6.1 三重验证机制

**1. 回归测试（原 PoC 重跑）**
- PoC 不再触发 crash → 漏洞修复确认，通过回归测试
- PoC 仍然触发 → 补丁无效，触发重试流程
- 额外运行 PoC 变体攻击（由 Exploiter Agent 生成），测试补丁是否存在绕过路径

**2. 副作用检测**
- 执行项目现有单元测试套件，测试通过率不低于补丁前基线
- 运行集成测试，验证核心业务流程不受影响
- 性能基准对比：补丁前后关键路径延迟变化不超过 5%

**3. 补丁安全扫描**
- 将生成的 diff 重新提交静态分析引擎扫描
- 确认补丁代码本身未引入新的 CWE 问题
- 特别检查补丁是否引入逻辑错误（off-by-one、条件判断反转）

### 3.6.2 重试与升级机制

| 场景 | 触发条件 | 处理动作 |
|------|----------|----------|
| 自动重试 | 补丁验证失败，重试次数 < 3 | 反馈失败原因给 Patcher Agent，重新生成补丁 |
| 方案升级 | 自动重试 3 次均失败 | 标记为"需人工处理"，通知安全团队 |
| 验证超时 | 验证流程超过 30 分钟 | 记录超时，转人工跟进 |
| 测试缺失 | 项目无单元测试 | 跳过副作用检测，标注"人工验证建议" |

## 3.7 漏洞管理层

### 3.7.1 漏洞生命周期状态机

```
[已发现] → [分析中] → [待验证] → [已确认] → [修复中] → [待审核] → [已修复] → [已关闭]
   |                                  |            ↑                              ↑
   |                                  |            └──[验证失败]重试(最多3次)──┘  |
   |                                  └──────[标记误报/接受风险]──────────────────┘
   └─ 所有状态均可查询历史操作记录

状态转移规则:
- 已确认 →[crash触发确认]→ 修复中
- 修复中 →[验证失败]→ 修复中（重试，计数+1）
- 修复中 →[重试>3次]→ 需人工处理
- 待审核 →[PR合并]→ 已修复
- 已修复 →[重扫通过]→ 已关闭
- 已确认/任意态 →[人工标记误报]→ 已关闭
```

| 状态 | 含义 | 进入条件 | 流出条件 |
|------|------|----------|----------|
| 已发现 | 静态/动态分析产出候选漏洞 | 分析引擎输出 | Agent 接手分析 |
| 分析中 | Agent 正在深度分析 | Planner 调度 | Exploiter 生成 PoC |
| 待验证 | PoC 已生成，等待沙箱执行 | PoC 生成完毕 | 沙箱执行结束 |
| 已确认 | 沙箱验证 PoC 有效 | crash 触发确认 | 进入修复流程 |
| 修复中 | 自动修复引擎生成补丁 | CVSS 评分完成 | 补丁验证结束 |
| 待审核 | 补丁验证通过，等待人工合并 | 验证三重通过 | PR 合并 |
| 已修复 | 补丁已合并，重扫确认 | PR 合并 | 重扫通过 |
| 已关闭 | 误报或接受风险 | 人工标记 | 终态 |

### 3.7.2 去重与优先级

- **去重规则**：相同 CWE 类型 + 相同代码位置（文件 + 函数）+ 相同根因 → 合并为同一漏洞记录
- **优先级计算**：`CVSS 分数 × 资产重要性权重 × 暴露面系数（内网服务 vs 公网接口）`
- **通知抑制**：相同漏洞在 72 小时内不重复发送告警，避免通知疲劳

# 6. 安全与合规设计

## 6.1 平台自身安全

- **网络隔离**：平台服务器部署于独立 VLAN，与办公网、生产网络隔离
- **PoC 加密存储**：所有生成的 PoC 使用 AES-256 加密存储，访问需双因素认证
- **权限最小化**：各组件使用最小权限运行，沙箱容器无法访问宿主系统
- **访问审计**：所有 PoC 查看、下载操作记录完整审计日志，保留 180 天
- **代码隔离**：扫描过程中的代码副本在扫描完成后自动清理

> ⚠ **关键安全原则**：平台产出的 PoC 具有实际攻击能力。PoC 库必须与代码仓库严格隔离，任何 PoC 的访问都需要审批，严禁将 PoC 发送给非授权人员。

## 6.2 数据安全与隐私

- 代码脱敏：使用云端 LLM API 时，对代码进行变量名混淆、字符串哈希替换处理
- 数据分级：核心业务代码（支付、鉴权等）只允许私有部署 LLM 处理
- 数据留存：漏洞记录根据严重程度保留 1~5 年；PoC 在漏洞修复确认后 30 天删除

## 6.3 合规考量

- **负责任披露**：发现涉及第三方开源组件的零日漏洞时，遵循 90 天负责任披露原则
- **CVE 申请**：对符合条件的漏洞，向 CVE 编号颁发机构（CNA）申请编号
- **审计合规**：漏洞发现、处理、修复的全流程记录可用于内部安全审计和合规报告

# 8. 风险与应对

| 风险 | 可能性 | 影响 | 应对措施 |
|------|--------|------|----------|
| LLM 幻觉导致误报过高 | 中 | 高 | 沙箱验证作为强制过滤器；误报数据用于持续微调 |
| 自动修复补丁引入新漏洞 | 中 | 高 | 补丁安全扫描强制执行；高危漏洞双重人工审核；补丁应用后重新全量扫描 |
| PoC 泄露导致平台成为攻击跳板 | 低 | 严重 | PoC 加密存储隔离；严格访问审计；定期安全评估平台自身 |
| 代码通过 LLM API 泄露 | 中 | 高 | 核心代码走私有部署 LLM；云端调用前代码脱敏处理 |
| Fuzzing 资源消耗过高 | 高 | 低 | 设置 CPU/内存配额；按优先级调度；低优先级任务利用离峰时段 |
| 研发团队抵触安全流程 | 中 | 中 | 低危漏洞全自动修复降低摩擦；安全报告可视化；与研发 KPI 挂钩 |
