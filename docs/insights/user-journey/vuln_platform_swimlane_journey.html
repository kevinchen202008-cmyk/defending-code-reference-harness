🧑‍💻 研发人员仓库 Owner / 研发 Lead
（安全团队可发起专项）
🙋 主动触发
  • 日常推送：Git Push / 创建·更新 MR。
  • 提交对象：亦可提交 代码库 / 软件包 / 在线服务 作为挖掘对象（安全团队可 API 发起专项）。
⏳ 离线静默
  • 任务交由 ① 服务层调度，研发无需干预、零摩擦，不改变工作流。
⏳ 离线静默
  • AI 挖掘引擎后台分析中，研发保持静默。
⏳ 动态等待
  • 沙箱已排除"幻觉"误报，无需人工复核。
⏳ 方案等待
  • 等待引擎生成并排序候选补丁。
⏳ 回归等待
  • 等待修复验证闭环（回归 / 副作用 / 安全扫描）跑完。
🙋 关键门控
  • 分级审核：低/中危 由研发 Owner/Lead approval；高危·严重 强制 安全团队 + Lead 双重审核。
  • 无自动合并：取消低危全自动合并，任何 AI 补丁合并前都需人工 approval。
🙋 合并决策
  • 消费漏洞：在 IDE 插件 / Web 看板查看验证报告与 diff 对比。
  • 一键合并：approval 后合并 PR（机器人无直接合并权，受分支保护）。
⏳ 闭环确认
  • 等待平台重扫判定根因是否彻底消除 → 状态「已关闭」。
🤖 AI 挖掘引擎① 漏洞扫描服务层 + ②~⑥
多实现体 · 可插拔
⚙️ 接入捕获
  • Git Hook / Pipeline CI 捕获变更并校验托管授权，登记为挖掘任务来源。
⚙️ ① 服务层调度
  • 任务生命周期：建任务、跟踪状态；按对象/等级指定引擎实现体（自研 / Codex 式 / Mythos·Glasswing 式）。
  • LLM 算力：分配 API 并发额度，保证算力打满、不闲置、不超预算。
⚙️ ②③ 分析挖掘
  • 静态 CodeQL/Semgrep + 动态 Syzkaller/AFL++ + 知识库 RAG。
  • Planner 调度 Researcher→Analyst→Exploiter，产出根因与最小 PoC。
⚙️ ④ 沙箱验证
  • 容器 + microVM 隔离（内核类走 VM、构建即沙箱）运行 PoC 触发 crash，映射 CWE，计算 CVSS。
⚙️ ⑤ 自动修复
  • Patcher 生成多候选 diff，按安全性/侵入性/置信度排序；多维度门控（CVSS×资产×侵入性×置信度）。
⚙️ ⑥ 修复验证
  • 回归（原 PoC + 变体）+ 副作用（单测 / 性能劣化 ≤5%）+ 独立工具链二次安全扫描。
  • 失败重试 ≤3 次转人工；含补丁回滚 / 熔断。
⚙️ 阻断门控
  • 对达门限缺陷下发审核工单、冻结自动合入，等待人工 approval。
⚙️ 通知与 PR
  • 将通过验证的补丁封装为 PR（机器人最小权限、签名提交），经 gitcode 待办/邮件 推送；收到 approval 后执行合并。
⚙️ ① 终审重扫
  • 合并后由 ① 漏洞扫描服务层重新下发全量扫描，确认根因消除，状态机流转「已关闭」。
优化要点（对齐 v3.2）：① 接入层升级为「漏洞扫描服务层」承接任务生命周期与 LLM 并发分配；引擎 ②~⑥ 为可插拔多实现体； 沙箱改容器 + microVM；取消低危全自动合并、合并前强制人工 approval；新增补丁回滚/熔断；合并后由服务层重新下发全量扫描（第 9 列回流）。
